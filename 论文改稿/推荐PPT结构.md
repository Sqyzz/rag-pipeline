  P3 研究背景：企业知识检索的核心矛盾

  这一页讲清楚问题，不要展开太多文献。

  重点表达：

  企业知识不同于普通网页文本，它通常具有三个特征：

  1. 细粒度事实密集：例如合同中的金额、义务、期限、责任主
     体。
  2. 跨条款关系复杂：一个风险判断往往依赖多个条款共同解
     释。
  3. 全局语义需要整合：管理者关心的不只是某一条款，而是合
     同整体风险、义务分布和责任链。

  因此企业知识检索不是简单的“找相似文本”，而是同时需要：

  - 找准局部事实；
  - 串联结构关系；
  - 汇总全局语义；
  - 保证证据可追溯。

  引出问题：传统 VectorRAG 缺少结构关系；原生 GraphRAG 偏
  全局社区摘要，容易弱化底层事实定位。

  P4 研究目标：为什么提出E-GraphRAG

  这一页用一句主线概括：

  本文希望解决的问题是：
  如何在企业文档中同时保持局部证据精确性、结构关系可推理
  性和全局语义整合能力。

  所以 E-GraphRAG 的设计不是单纯“加图谱”，而是围绕企业知
  识架构进行两类改造：

  1. 底层：用四层结构重新组织企业知识。
  2. 顶层：用 Agentic Decomposer 把用户问题映射到合适的检
     索层级。

  这页可以放一张总览图：
  企业文档 → 四层知识图谱 → Query拆解 → Local/Structural/
  Global路由 → 证据整合 → 回答生成

  P5 为什么要分四层结构

  这一页是重点。

  四层结构建议这样讲：

  1. 属性层
     承载最细粒度的事实，例如合同金额、期限、义务、违约责
     任、适用条件。
     对应企业检索中的“查具体事实”。
  2. 实体层
     承载合同主体、条款对象、业务对象等核心实体。
     用于把分散事实挂接到稳定对象上，避免只靠文本块召回。
  3. 关键词/关系层
     承载实体之间的关系、主题词和中观语义连接。
     用于支持跨条款、跨片段的结构推理。
  4. 社区层
     承载宏观主题聚合，例如风险类型、义务群组、争议解决机
     制等。
     用于支持合同整体分析、风险总结、主题归纳。

  这里要强调设计理由：

  企业知识天然是多粒度的。
  如果只保留文本块，系统能召回但不能推理；如果只依赖社区
  摘要，系统能概括但容易丢失细节。四层结构的目的就是把企
  业知识拆成 “微观事实 - 业务实体 - 结构关系 - 宏观主题”
  四个层级，使不同类型的问题能进入不同的证据空间。

  P6 四层结构如何和Agentic Decomposer联动

  这一页讲核心机制。

  Agentic Decomposer 的作用不是回答问题，而是把用户 Query
  转换成结构化检索计划。

  例如用户问：

  “这份合同中有哪些可能影响公司后续履约风险的条款？”

  系统不会直接全文检索，而是先拆解为：

  - 是否存在违约责任条款？
  - 是否存在终止条件？
  - 是否存在保密义务？
  - 是否存在责任限制？
  - 这些条款之间是否构成风险联动？

  然后给每个子问题分配路由：

  - 查具体条款 → Local Route → 属性层/实体层；
  - 查条款之间关系 → Structural Route → 关系层；
  - 查整体风险画像 → Global Route → 社区层。

  可以用一句话总结：

  四层图结构提供“可检索的知识空间”，Agentic Decomposer 决
  定“用户问题应该进入哪一层、以什么顺序检索”。

  P7 为什么要这样设计

  这一页解释设计必要性。

  核心逻辑：

  企业用户的问题通常不是单一类型，而是复合型问题。
  例如“这份合同有没有风险”看似是全局问题，但实际上需要先
  定位底层条款，再识别条款关系，最后形成整体风险判断。

  如果直接走全局摘要，可能遗漏关键条款；
  如果只走局部检索，又无法形成风险判断；
  所以需要 Agentic Decomposer 先判断问题意图，再动态组合
  Local、Structural、Global 三类检索路径。

  这就是本文从固定检索管线转向规划驱动检索的原因。

  P8 如何确保用户Query需求被正确映射

  这一页回答你提到的“整个过程怎么确保用户query的需求正确
  被映射”。

  可以从三层保障讲：

  1. 任务类型约束
     预先把企业知识问题划分为 Local、Structural、Global
     三类能力需求。用户 Query 先被映射到这三类任务，而不
     是直接进入检索器。
  2. Schema约束
     Agentic Decomposer 拆解问题时，不是自由生成，而是结
     合合同 Schema、实体类型、关系类型和条款类型生成结构
     化检索需求。
  3. 路由字段约束
     每个子问题必须生成 route_type、semantic_need、
     anchors、target_patterns 等字段。
     这些字段会被后端编译成可执行检索动作，避免自然语言问
     题和检索层之间脱节。

  一句话总结：

  用户需求的正确映射依赖“任务分类 + Schema约束 + 结构化路
  由字段”三重约束。

  P9 数据集怎么构建，为什么符合研究场景

  这一页讲实验数据构建流程。

  建议这样组织：

  本文选择 CUAD 合同数据集作为企业知识检索实验语料，原因
  是：

  1. 合同是典型企业知识载体；
  2. 条款结构清晰，适合局部事实检索；
  3. 条款之间存在义务、责任、风险等复杂关系，适合结构推
     理；
  4. 合同整体风险判断需要全局语义整合，适合验证 GraphRAG
     能力。

  构建流程：

  1. 原始合同文档收集
     选取 CUAD 合同文本作为企业文档语料。
  2. 文档切片与预处理
     按合同条款、段落结构进行切分，保留原始 Chunk 作为证
     据来源。
  3. Schema约束知识抽取
     抽取实体、属性、关系，并映射到预设合同 Schema。
  4. 分层图谱构建
     将抽取结果组织为属性层、实体层、关键词/关系层、社区
     层。
  5. 测试问题构建
     围绕 Local、Structural、Global 三类能力构建评测任
     务，覆盖单条款查询、条款解释、多条款联动和合同风险审
     查。

  这里要强调：

  数据集不是随便拿来做问答，而是围绕企业知识检索的三类核
  心需求构建，因此能验证本文框架的真实适配性。

  P10 实验结果：只讲最关键的三点

  10分钟内不要铺所有表格，只挑核心指标。

  建议讲三条：

  1. 局部事实检索更准
     E-GraphRAG 的 Context Precision 达到 0.8765，背景噪
     声比例由 35.6% 降至 12.3%。
     说明四层结构和 Local 路由能有效提升条款级事实定位能
     力。
  2. 结构关系推理更稳
     在多条款联动任务中，系统通过 Structural Route 和路径
     约束减少无关证据，提升证据链可信度。
  3. 全局语义整合更可靠
     Global 任务中 Semantic Similarity 达到 0.9381，说明
     社区层与证据下钻机制能够支撑合同整体风险分析。

  P11 研究结论

  总结成三句话：

  1. 企业知识检索的关键不是单纯提升召回，而是让不同粒度的
     问题进入匹配的知识层级。
  2. 四层图结构解决的是企业知识的组织问题，使微观事实、结
     构关系和宏观主题能够统一表达。
  3. Agentic Decomposer 解决的是用户 Query 到检索路径的映
     射问题，使系统能够按问题意图动态调度 Local、
     Structural 和 Global 检索。

  最后落到价值：

  E-GraphRAG 更适用于合同审查、制度问答、企业合规分析等低
  容错、高可追溯的知识服务场景。
