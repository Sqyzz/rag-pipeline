# Pipeline 对比原则（VectorRAG vs KG-RAG vs GraphRAG）

## 总目标

本实验旨在比较三类检索增强生成系统在企业知识管理场景中的能力边界：

- **VectorRAG**：语义检索驱动的局部问答
- **KG-RAG**：结构化图遍历驱动的多跳推理
- **GraphRAG**：社区层级摘要驱动的全局综合

实验设计的核心要求是：

> 不强行让三者流程一致，而是保证底层资源一致、预算透明、对比公平。

------

------

# 原则 1：共享底盘（Shared Backbone Consistency）

三种方法必须共享相同的数据与基础组件，否则对比无意义。

### 必须统一的部分：

- **相同数据集与预处理**
  - 同一份 sampled corpus
  - 相同去重策略
  - 相同文本清洗规则
- **相同 chunking 策略**
  - chunk_size、overlap 完全一致
- **相同 embedding 模型**
  - VectorRAG 的 dense retrieval
  - KG-RAG 的证据回填
  - GraphRAG 的社区文本表示
     都必须来自同一 embedding
- **相同最终生成模型（Answer LLM）**
  - 三者最后的 answer synthesis 使用同一个 LLM（本地或 API）
- **统一输出格式**
  - answer
  - retrieved evidence contexts
  - token usage metadata

📌 目的：

> 保证性能差异来自方法结构，而不是数据或模型差异。

------

------

# 原则 2：允许方法本体不同（Method-Defining Steps Are Preserved）

公平对比不是“步骤数量相同”，而是：

> 每个方法必须保留其定义性的核心机制。

### VectorRAG 的核心机制：

- dense retrieval (top-k)
- single-step synthesis

### KG-RAG 的核心机制：

- entity linking（确定 traversal 起点）
- multi-hop traversal（1–3 hops）
- subgraph evidence aggregation

### GraphRAG 的核心机制：

- community detection（Leiden）
- community summaries（hierarchical compression）
- map-reduce global synthesis

📌 结论：

> KG 的 linking/traversal 与 Graph 的 map/reduce 不是“额外增强”，而是方法定义的一部分，不能删掉。

------

------

# 原则 3：对比必须分为两种模式（Two Complementary Comparison Regimes）

为了避免“某方法靠更多 LLM 调用堆出来”的质疑，实验必须同时报告两类对比：

------

## 3.1 Best-Effort Setting（方法上限对比）

每个方法按其原生设计完整运行：

- KG-RAG 使用 traversal
- GraphRAG 使用 map/reduce
- VectorRAG 使用 top-k retrieval

回答的问题是：

> “方法在合理实现下的能力上限是什么？”

------

## 3.2 Budget-Matched Setting（公平预算对比）

所有方法在同一 query-time 计算预算下运行，例如：

- 总输入 tokens ≤ 3000
- 总 LLM 调用次数 ≤ 2
- 总输出 tokens ≤ 800

回答的问题是：

> “在相同成本约束下，谁更划算？”

📌 这一步是实验公平性的关键保障。

------

------

# 原则 4：任务必须分型评测（Task-Type Stratification）

单一 benchmark 无法同时体现三者优势，因此 query 必须按能力分组：

------

## Task A：Multi-hop factual reasoning（KG 主场）

- 问题形式：实体关系推理、多跳链路
- 预期：KG-RAG 优势明显

------

## Task B：Global sensemaking & synthesis（Graph 主场）

- 问题形式：跨主题总结、全局风险归纳
- 预期：GraphRAG 优势明显

------

## Task C：Local factual retrieval（Vector 主场）

- 问题形式：局部事实定位、精确问答
- 预期：VectorRAG 最稳定

📌 目标：

> 不追求单一胜负，而是刻画方法能力边界。

------

------

# 原则 5：必须报告完整成本（Cost Transparency）

三者复杂度不同，因此必须显式记录：

### Query-time metrics

- LLM 调用次数
- prompt tokens
- completion tokens
- latency (p50/p95)

### Indexing-time metrics

- triple extraction cost（KG/Graph）
- community summary cost（Graph）
- index build time（Vector）

📌 避免结论变成：

> “GraphRAG 更好，因为它花了十倍 token。”

------

------

# 原则 6：共享图构建，分叉在使用方式（Shared Graph, Divergent Usage）

KG-RAG 与 GraphRAG 可以共享同一底层知识图谱：

- triples.jsonl
- graph.pkl

分叉点在 query-time：

- KG-RAG：traversal over triples
- GraphRAG：community summary abstraction

📌 这保证了：

> 图质量一致，对比更公平。

------

------

# 原则 7：消融实验必须解释贡献（Ablation for Mechanistic Understanding）

GraphRAG 与 KG-RAG 必须做关键组件消融：

### GraphRAG Ablations

- no community summaries
- no hierarchy
- no shuffle
- no map-reduce

### KG-RAG Ablations

- hop=1 vs hop=2 vs hop=3
- no entity linking（退化检索）
- traversal pruning

📌 目的：

> 证明提升来自机制，而不是偶然参数。
